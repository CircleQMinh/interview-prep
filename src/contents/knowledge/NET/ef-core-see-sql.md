## Overview

In an EF Core application, there are several **clean and production-safe ways** to see the SQL being executed at runtime.

This is essential for:

- Debugging LINQ queries
- Detecting N+1 problems
- Performance tuning
- Production diagnostics

---

##  Enable EF Core Logging (Recommended)

The most common approach is enabling logging via `ILogger`.

### ASP.NET Core (`Program.cs`)

```csharp
builder.Services.AddDbContext<AppDbContext>(options =>
{
    options.UseSqlServer(builder.Configuration.GetConnectionString("Default"))
           .EnableDetailedErrors()
           .EnableSensitiveDataLogging() // Development only
           .LogTo(Console.WriteLine, LogLevel.Information);
});
````

### What This Does

* Logs generated SQL
* Logs parameters (when `EnableSensitiveDataLogging()` is enabled)
* Logs execution timing
* Integrates with ASP.NET Core logging pipeline

---

### ⚠ Important

`EnableSensitiveDataLogging()` should **NOT** be enabled in production.

It logs:

* Parameter values
* Potentially sensitive data (passwords, tokens, PII)

Use only in development environments.

---

##  Use `appsettings.json` Logging Configuration (Cleaner Approach)

Instead of modifying code, configure logging via `appsettings.json`.

```json
{
  "Logging": {
    "LogLevel": {
      "Default": "Information",
      "Microsoft.EntityFrameworkCore.Database.Command": "Information"
    }
  }
}
```

This logs SQL commands without changing `DbContext` configuration.

Example output:

```
Executed DbCommand (12ms) [Parameters=[@__id_0='1'], CommandType='Text']
SELECT [u].[Id], [u].[Name]
FROM [Users] AS [u]
WHERE [u].[Id] = @__id_0
```

This is typically the **best standard approach**.

---

##  Use `ToQueryString()` (Inspect a Specific Query)

To inspect SQL generated by a LINQ query **without executing it**:

```csharp id="z2mp4s"
var query = _context.Users.Where(u => u.Id == 1);

string sql = query.ToQueryString();

Console.WriteLine(sql);
```

Important notes:

* Does NOT execute the query
* Only shows generated SQL
* Available in EF Core 5+

Very useful for debugging complex LINQ expressions.

---

##  Use SQL Server Profiler / Extended Events

To see all SQL reaching SQL Server, use external database tools:

* SQL Server Profiler
* Extended Events
* Azure SQL Query Performance Insight
* SQL Server Query Store

These tools provide:

* Exact SQL text
* Execution duration
* CPU usage
* Deadlocks
* Blocking information
* Execution plans

Ideal for production diagnostics.

---

##  Use EF Core Interceptors (Advanced / Professional)

For structured logging or telemetry, implement a `DbCommandInterceptor`.

```csharp id="w9ka1t"
public class SqlLoggingInterceptor : DbCommandInterceptor
{
    public override InterceptionResult<DbDataReader> ReaderExecuting(
        DbCommand command,
        CommandEventData eventData,
        InterceptionResult<DbDataReader> result)
    {
        Console.WriteLine(command.CommandText);
        return base.ReaderExecuting(command, eventData, result);
    }
}
```

Register it:

```csharp
options.AddInterceptors(new SqlLoggingInterceptor());
```

Use cases:

* Custom telemetry
* Performance monitoring
* Auditing
* Integration with Application Insights
* Structured logging pipelines

---

##  Log Only Slow Queries (Better Production Strategy)

Instead of logging everything:

* Use interceptors with a `Stopwatch`
* Log only when execution time exceeds a threshold
* Use Application Insights dependency tracking
* Use SQL Server Query Store

This avoids excessive log noise and improves observability quality.

---

## Comparison Table

| Method                     | Use Case                      | Production Safe |
| -------------------------- | ----------------------------- | --------------- |
| `LogTo()`                  | Quick debugging               | Dev only        |
| `appsettings.json` logging | Standard logging solution     | Yes             |
| `ToQueryString()`          | Inspect specific LINQ query   | Yes             |
| SQL Profiler / Query Store | Database-level monitoring     | Yes             |
| Interceptors               | Custom monitoring & telemetry | Yes             |

---

## Recommended Strategy

### Development

* Enable EF Core logging via `appsettings.json`
* Use `EnableSensitiveDataLogging()` only locally
* Use `ToQueryString()` for debugging LINQ

### Production

* Use Application Insights dependency tracking
* Use SQL Server Query Store
* Use interceptors for structured telemetry
* Avoid logging sensitive data
* Prefer logging slow queries instead of all queries

---

## Summary

EF Core provides multiple ways to inspect generated SQL:

* Built-in logging
* Configuration-based logging
* Query inspection via `ToQueryString()`
* Interceptors
* Database-level tools

Choose the approach based on environment and observability needs.
